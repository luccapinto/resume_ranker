# Card 1.2 — Implementação do `PIIRedactor` e Reconhecedores BR

Tag: 🤖 Execução Autônoma

## Contexto
Por design de governança de privacidade de dados, nenhuma informação pessoal identificável (PII) pode ser transmitida para APIs externas (LLMs ou provedores de embedding baseados em nuvem). A anonimização deve ocorrer de forma estritamente local no backend antes de qualquer outra etapa do pipeline de processamento.

## Objetivo
Implementar o módulo `PIIRedactor` local usando Microsoft Presidio e spaCy, com suporte nativo a formatos brasileiros de identificadores (CPF, RG, e formatos locais de telefone).

## Especificação Técnica
- **Motor Principal**:
  - Utilizar a biblioteca `presidio-analyzer` e `presidio-anonymizer` da Microsoft.
  - Carregar o modelo de NLP do spaCy `pt_core_news_lg` para identificação de entidades em português (como nomes de pessoas e locais).
- **Reconhecedores Brasileiros Customizados**:
  - Implementar um reconhecedor customizado (herdando de `EntityRecognizer` ou usando `PatternRecognizer`) para CPF (verificando formato `xxx.xxx.xxx-xx` e dígitos limpos).
  - Implementar reconhecedor para RG (registro geral) brasileiro com regex flexível.
  - Implementar reconhecedor para telefones brasileiros (DDDs de 2 dígitos + celulares com 9 dígitos ou fixos com 8 dígitos, considerando ou não hifens/parênteses).
- **Processamento de Anonimização**:
  - O `PIIRedactor` deve substituir entidades sensíveis (`PERSON`, `EMAIL`, `PHONE_NUMBER`, `CPF`, `RG`, `IP_ADDRESS`, `LOCATION`, `ORGANIZATION`) por placeholders estruturados, ex: `[NOME_REDACT_1]`, `[CPF_REDACT_1]`.
  - Manter um dicionário/mapa reversível das redações de forma efêmera em memória (se necessário para re-identificação posterior nas explicações), mas o texto salvo no Postgres para auditoria deve ser o texto anonimizado (ou as entidades mapeadas de forma segregada e segura). *Para fins de simplicidade e segurança, o texto anonimizado é o principal a ser trafegado.*
  - Garantir tratamento de exceção robusto para que, se o redator falhar por qualquer motivo, o processamento do currículo/vaga seja interrompido (fail-fast).

## Critérios de Aceite
- [ ] O `PIIRedactor` inicializa com sucesso o modelo `pt_core_news_lg`.
- [ ] O texto contendo nomes ("João da Silva"), e-mails ("joao@provedor.com"), telefones ("(11) 98765-4321") e CPFs ("123.456.789-00") é devidamente anonimizado, substituindo os valores reais por tokens (ex: `[PERSON]`, `[EMAIL]`, etc.).
- [ ] Entidades brasileiras complexas (RG, CPF com ou sem máscara) são identificadas com precisão e taxa de falso negativo próxima a zero.
- [ ] O módulo de anonimização roda em menos de 500ms por página de currículo sob condições locais comuns.

### Testes Automatizados
- [ ] Testes unitários com pelo menos 15 currículos/textos sintéticos simulados contendo diversas combinações de PII brasileiras para validar a taxa de detecção (Precision e Recall mínimos de 95% para CPF e e-mail).
- [ ] Garantir que o pipeline de ingestão lança exceção em caso de erro na inicialização do analisador de PII.
