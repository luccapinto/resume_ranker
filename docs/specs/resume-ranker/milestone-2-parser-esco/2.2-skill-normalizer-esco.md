# Card 2.2 — Implementação do `SkillNormalizer` com Taxonomia ESCO

Tag: 🤖 Execução Autônoma

## Contexto
Competências técnicas e comportamentais extraídas por LLMs podem vir com variações linguísticas, abreviações ou sinônimos diferentes (ex: "JS", "Javascript", "Ecmascript"). Para termos uma comparação precisa e robusta na busca estruturada e híbrida, precisamos mapear cada competência para a taxonomia padrão europeia **ESCO** (European Skills, Competences, Qualifications and Occupations).

## Objetivo
Implementar o componente `SkillNormalizer` para mapear de forma canônica as competências extraídas contra o dicionário oficial da ESCO, utilizando técnicas de correspondência exata, fuzzy matching e fallback por embeddings locais.

## Especificação Técnica
- **Taxonomia ESCO (Offline)**:
  - Importar a versão da base ESCO de competências (CSV ou JSON) para o projeto.
  - Carregar em memória ou em uma tabela de busca rápida (SQLite ou Postgres indexado) as colunas contendo `conceptUri`, `preferredLabel` e `altLabels` (sinônimos).
- **Algoritmo de Correspondência do `SkillNormalizer`**:
  - **Fase 1 (Match Exato)**: Limpar a string da skill extraída (lowercase, remoção de caracteres especiais) e buscar correspondência exata contra o `preferredLabel` ou sinônimos (`altLabels`).
  - **Fase 2 (Fuzzy Match)**: Caso não encontre match exato, utilizar Levenshtein/RapidFuzz para identificar correspondências aproximadas com score superior a um threshold definido (ex: 85%).
  - **Fase 3 (Fallback por Embedding)**: Se o fuzzy match falhar, utilizar um modelo de embedding local (ex: sentence-transformers) para computar a similaridade vetorial entre a skill extraída e as skills indexadas na taxonomia ESCO. Escolher a skill ESCO com maior similaridade acima de um threshold mínimo (ex: 0.75).
- **Bucket Não Mapeado**:
  - Competências que não passarem em nenhum dos filtros acima devem ser direcionadas para uma coleção de "skills não mapeadas".
  - Logar estas ocorrências de forma que o sistema mantenha a rastreabilidade e exponha esses itens na interface, em vez de descartá-los silenciosamente.

## Critérios de Aceite
- [ ] O `SkillNormalizer` inicializa corretamente a base offline da ESCO e o modelo de fallback por embedding.
- [ ] Variações comuns ("Python 3", "Py", "Javascript", "JS") são mapeadas corretamente para suas URIs canônicas da ESCO correspondentes (ex: `http://data.europa.eu/esco/skill/python`).
- [ ] Termos sem qualquer correlação lógica (ex: "abracadabra") não são mapeados e entram no bucket de "não mapeados" com log detalhado de falha.
- [ ] Performance do mapeamento em lote de um currículo de 20 skills roda em menos de 2 segundos localmente.

### Testes Automatizados
- [ ] Criar testes unitários com uma lista conhecida de skills informais e validar se o normalizador retorna as URIs esperadas.
- [ ] Testar cenários de concorrência e garantir que a base offline é thread-safe.
